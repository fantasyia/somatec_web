## [4] (HIGH) [components-public] Turnstile captcha never initializes on component remount → legit form submits rejected with 400
- file: src/components/forms/fields/TurnstileWidget.tsx : 41, 57-69, 79-84
- desc: Rendering of the Turnstile widget is gated entirely on `scriptLoaded`, which is only set by the `<Script onLoad>` callback (line 83). The `<Script>` has no `id`, so Next.js uses the script `src` as its cacheKey. Next's `loadScript` (node_modules/next/dist/client/script.js lines 73-75) bails out early WITHOUT calling `onLoad` whenever `LoadCache.has(cacheKey)` is already true — i.e. on every mount AFTER the first. Next only re-fires `onReady` on remount (lines 204-208), not `onLoad`, and this component uses `onLoad`. So on any remount of TurnstileWidget after the Cloudflare script has loaded once, `setScriptLoaded(true)` never runs, `render()` is never called, no widget is created, and `onToken` is never invoked — `captchaToken` stays ''. The server (src/lib/turnstile/verify.ts lines 43-45) returns `{ ok:false, reason:'missing_token' }` (NOT infraFailure) for an empty token when TURNSTILE_SECRET_KEY is set, so the route (src/app/api/forms/submit/route.ts lines 107-119) rejects with 400 'Validação de segurança falhou'.
- impact: In production, the second (and every subsequent) submission of any form whose TurnstileWidget remounted fails with HTTP 400. SampleRequestModal is the worst case: it mounts the widget inside `{open && ...}`, so open→close→open→submit always fails the captcha. ContactForm/RepresentanteForm hit it on client-side navigation between two form pages (component remounts, script already in LoadCache). Silent lead loss on a production lead-capture site.
- fix: No useEffect do TurnstileWidget, nao depender de onLoad para remount: `useEffect(() => { if (window.turnstile || scriptLoaded) render(); ... }, [scriptLoaded, render])`. Em remount o script ja esta no DOM/LoadCache, entao window.turnstile existe e render() pode rodar na hora. Alternativa/complemento robusto: dar um `id` estavel ao `<Script>` e trocar `onLoad` por `onReady` (que o Next re-dispara em remount via o effect hasOnReadyEffectCalled quando LoadCache.has(cacheKey)). Recomendo tambem resetar captchaToken para '' ao fechar/abrir o modal e apos cada submit, e usar window.turnstile.reset/execute para emitir um token fresco, ja que o token e de uso unico (evita o caso stale-token tambem no 1o open seguido de reabertura).
